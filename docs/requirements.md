# FobForge — Requirements Document

**Version:** 1.0  
**Date:** 2026-05-07  
**Author:** MRGN79  
**Status:** Approved

---

## 1. Project overview

FobForge is a browser-based tool that allows field technicians to manage RFID
badges in BTicino/Legrand Sfera New & Robur door entry systems by editing
`.prj` files directly, without requiring a physical connection to the reader
hardware.

### 1.1 Problem statement

BTicino's official software (TiSferaDesign) requires a USB connection to the
physical RFID reader module in order to register new badges. This forces
technicians to visit the installation site with a laptop every time a new badge
needs to be added.

### 1.2 Proposed solution

FobForge allows technicians to add badge UIDs manually (the UID code is printed
on every transponder) by editing the `.prj` file directly in the browser.

---

## 2. Scope

### 2.1 In scope (v1 — MVP)

- Load an existing `.prj` file generated by TiSferaDesign
- Display the list of contacts and residents
- Create a new RFID badge by entering its UID manually
- Assign a badge to a contact
- Remove a badge from a contact
- Save the modified `.prj` file

### 2.2 Out of scope (v1)

The following features are explicitly excluded from v1 and deferred to v2:

- Create a new `.prj` file from scratch
- Add, edit or delete contacts/residents
- Bulk import from CSV
- Export to CSV or PDF
- NFC badge reading via WebNFC API
- Biometric data management
- Firmware update functionality
- Module configuration

---

## 3. User profile

**Primary user:** Field technician / installer working with BTicino Sfera
door entry systems.

**Technical level:** Familiar with the `.prj` file concept and TiSferaDesign
software, but treats the file format as a black box. No programming knowledge
required.

**Working context:** May be on-site without access to the reader hardware.
Works on Windows PC, but the tool must also be usable on tablet with keyboard
and mouse.

## 4. Functional requirements

### FR-01 — Load .prj file
- The user can load a `.prj` file via drag & drop or file picker
- The file is read entirely client-side (no server upload)
- If the file is invalid or unreadable, a system error toast is displayed

### FR-02 — Display contacts
- After loading, the tool displays the full list of contacts/residents
- For each contact, the following is shown:
  - Name and surname
  - Apartment number
  - SCS address
  - Number of badges currently assigned

### FR-03 — Create new badge
- The user can enter a badge UID manually
- The UID is validated via a centralized `validateUID()` function
- Validation rules:
  - Exactly 8 characters
  - Hexadecimal only (0-9, A-F)
  - Automatically converted to uppercase
  - Must not already exist in the project
- If validation fails, an inline error message is shown
- If the UID already exists in the project, an inline error is shown

### FR-04 — Assign badge to contact
- The user can assign an existing badge to a contact
- If the badge is already assigned to that contact, an inline error is shown
  and the operation is not performed

### FR-05 — Remove badge from contact
- The user can remove a badge assignment from a contact
- The badge is also removed from the global badge list if it has no other
  assignments

### FR-06 — Save .prj file
- The user can save the modified `.prj` file
- The output file maintains full compatibility with TiSferaDesign
- The file is downloaded directly to the user's device
- No data is sent to any server at any point

## 5. Non-functional requirements

### NFR-01 — Privacy
- The `.prj` file never leaves the user's device
- No analytics, no tracking, no external API calls

### NFR-02 — Compatibility
- Must work on Chrome (desktop and Android)
- Must work on Firefox and Edge (desktop)
- Must be usable on tablet with external keyboard and mouse

### NFR-03 — Internationalization
- Interface available in English and Spanish
- Language selector visible in the UI
- All user-facing strings managed via a centralized i18n system
- No hardcoded UI text outside the i18n module

### NFR-04 — Security
- Cryptographic keys used for `.prj` file handling must not appear
  in plain text in the source code or repository

### NFR-05 — Performance
- The tool must load and parse a `.prj` file in under 3 seconds
  on a mid-range device

### NFR-06 — Zero dependencies (except sql.js)
- No frontend frameworks (no React, Vue, Angular)
- No CSS frameworks (no Bootstrap, Tailwind)
- No build tools required
- Single external dependency: sql.js (SQLite compiled to WebAssembly)

## 6. Error handling

| Situation | Behavior |
|-----------|----------|
| Invalid or corrupt `.prj` file | System toast error |
| UID with wrong length | Inline error on field |
| UID with invalid characters | Inline error on field |
| UID already exists in project | Inline error on field |
| Badge already assigned to contact | Inline error on field |
| Save fails (disk full, etc.) | System toast error |

## 8. Format research notes

The following was discovered through binary analysis of real `.prj` files
and the `GUIDirector.exe` executable:

- A `.prj` file is a **ZIP archive** renamed to `.prj`
- It contains three files: `0.xtz`, `DCDB.xtz`, `1.xtz`
- Each `.xtz` file is itself an encrypted ZIP (ZipCrypto)
- `DCDB.xtz` contains `Device_contacts.db3`, a standard SQLite3 database
- The SQLite database contains the following tables:
  - `MEMBER` — contacts / residents
  - `APT` — apartments
  - `MEMBER_APT` — contact ↔ apartment assignments
  - `BADGE` — RFID badges (UID as 8-char hex string)
  - `MEMBER_BADGE` — contact ↔ badge assignments
  - `MEMBER_LOCK_PASSWORD` — numeric access codes
  - `MEMBER_MINUTIAE` — biometric data (fingerprints)
  - `MEMBER_MINUTIAE_SCAN` — biometric scan data
  - `DB_INFO` — database metadata
- Encryption uses ZipCrypto (legacy ZIP encryption, not AES)
- Keys are hardcoded in `GUIDirector.exe`
- No EULA was found in the software — no reverse engineering clause exists
